01 — WHAT TO SHARE

Three things, that's it.

Frontend

A staging URL (or a build we can host) the agents can drive. Optional repo for deeper context, and a test account to log in.

Backend

Your API base + OpenAPI/GraphQL spec — or a container / docker-compose we spin up in the sandbox. Source repo optional, for tracing.

Data

A schema plus an anonymized or synthetic seed so flows have realistic state. We require no real PII.

02 — THE TEMPLATE

One file describes it all.

One matraix.yaml is the entire handoff. It points us at your frontend, backend, and data, defines the input each flow expects, and declares the output you want back — so the report drops straight into your pipeline. Copy the file below, fill in your values, and send it over (or drop it in your repo root). We'll walk through every block underneath.

Required the minimum to start a run Optional sharpens results, never blocks
matraix.yaml
# matraix.yaml — describe your stack so the agents can run it
project: acme-checkout                  # slug we use to name your sandbox
contact: "eng@acme.com"                 # who we ping with questions

# ── 1. The surface the agents click through ──────────────
frontend:
  url: "https://staging.acme.com"      # reachable HTTPS URL the agents drive
  repo: "github.com/acme/web"          # optional, for deeper context
  auth:                              # how agents sign in
    type: test-account
    user: "bot@acme.com"
    secret: env:MATRAIX_PW            # reference a secret — never paste it

# ── 2. The API behind every click (the I/O contract) ─────
backend:
  api:     "https://staging.acme.com/api"
  spec:    "openapi.yaml"              # OpenAPI 3 / GraphQL SDL — request+response shapes
  sandbox: "docker-compose.yml"        # or a container image we spin up
  repo:    "github.com/acme/api"       # optional source for tracing failures

# ── 3. Enough state to make flows realistic ──────────────
data:
  schema: "schema.sql"               # table / model definitions (.sql, prisma, JSON Schema)
  seed:   "anonymized_sample.csv"     # synthetic or anonymized rows (.csv / .json / dump)
  pii:    none                        # we require no real PII

# ── 4. What a successful run looks like, end to end ──────
goals:
  - flow: checkout
    entry:   "/cart"                 # where the agent starts
    success: order_confirmed          # event/state that counts as a win
    inputs:                          # data the agent supplies, WITH format
      - { field: card,         type: string, format: "test PAN, e.g. 4242 4242 4242 4242" }
      - { field: shipping_zip, type: string, format: "5-digit US ZIP" }
    output:                          # what you want captured per attempt
      schema:  "order.schema.json"     # JSON Schema of the result we record
      metrics: [conversion rate, latency_ms, error rate]

# ── 5. Who the agents emulate ───────────────────────────
segments: [mobile, new_users, non-native English]

# ── 6. Where the report lands in your pipeline ──────────
report:
  format:  [json, html]            # machine-readable + human-readable
  webhook: "https://acme.com/hooks/matraix"  # we POST results here when done

Block by block

Six blocks. Here's what each one is for, and the exact format we expect.

project / contactRequired

A short slug we use to name and isolate your sandbox, plus a human to reach when something's ambiguous.

Format — project: kebab-case string. contact: any email.
frontendRequired

The surface the agents actually click through. Give us a reachable staging URL (or a build we can host) and an auth block so the agents can sign in like a real user.

Format — url: HTTPS URL. auth.type: test-account, magic-link, or none. Pass credentials by reference — env:VAR_NAME or a shared secret link — never paste real secrets into the file.
backendRecommended

So we can read and verify the I/O behind each click, not just the pixels. The spec is the contract: it tells us every endpoint's request and response shape, so the agents send well-formed input and we can grade the output precisely. No spec? A reachable API plus the staging URL still works as a black box.

Format — spec: OpenAPI 3 (YAML/JSON) or GraphQL SDL. sandbox: a docker-compose.yml or image ref (ghcr.io/acme/api:tag) if you'd rather we run it isolated.
dataRequired

Enough state that flows behave like production — products in a catalog, a user with a cart, etc. A schema plus a small seed is plenty.

Format — schema: .sql, Prisma/ORM models, or JSON Schema. seed: .csv, .json, or a DB dump — synthetic or anonymized. pii: none is the expected value; we require no real PII.
goalsRequired — the heart of the file

This is where you define the input/output contract the agents are graded against. Each flow names where it starts (entry), what counts as a win (success), the inputs the agent must supply (each with a type and human-readable format), and the output you want captured per attempt — a result schema plus the metrics that matter.

Input format — a list of { field, type, format } so agents generate valid, realistic values. Output format — output.schema points at a JSON Schema for the result we record; output.metrics is any of our 900+ metrics. Add as many flow entries as you have journeys.
segmentsOptional

The personas to weight the simulated population by, so results are sliced the way you think about users. Skip it and we run a balanced default population.

Format — a list of tags (mobile, new_users, enterprise, non-native English…) or named segments we map onto the persona dimension space.
reportOptional

Where the results should go. Pick the output formats you want and give us a webhook and the report lands in your pipeline automatically — no portal round-trip.

Format — format: any of json, html, jsonl. webhook: an HTTPS endpoint we POST to on completion. Omit it and you pull results from the dashboard instead.

What you get back

Two artifacts per run — one for your team to read, one for your machines to consume.

←report.json / .html

  • A segmented A/B report — friction, conversion lift, and failure points sliced by your segments
  • Per-flow metrics exactly as you named them in goals[].output.metrics
  • A funnel friction map with the step where each segment drops off
  • Ranked findings with severity, so you know what to fix first

←trajectories.jsonl

  • Every agent run as obs → action → reward steps — full session replays you can audit
  • Results validated against your output.schema, so they slot straight into your stores
  • SFT / RL-ready trajectories for training, one JSON object per line
  • Delivered to your report.webhook, or pulled via the dashboard / API
03 — HOW IT WORKS

From your repo to results.

01Share

Send your matraix.yaml — frontend URL, backend spec, and an anonymized data seed. That's all we need to start.

02Sandbox

We stand up an isolated, per-customer sandbox of your stack — network-restricted, no production access, torn down after the run.

You get a segmented report — friction, lift, and failures by segment — plus full replays and training-ready trajectories.

04 — YOUR CODE & DATA

Where it runs, and who can see it.

Q01Where does our code run?+
In an isolated, per-customer sandbox — network-restricted and never co-mingled with other customers. It's torn down after the run, and you can request deletion at any time.
Q02Do you need real production data?+
No. A schema plus an anonymized or synthetic seed is enough to give flows realistic state. We require no real PII, and any inputs you share are handled under NDA.
Q03Who can access it, and can we revoke it?+
Scoped test credentials only, encrypted in transit and at rest, accessible to your assigned team under NDA. Access is revocable and deletable on request, and your code is never used to train shared models without explicit opt-in.
Q04What if we can't share source code?+
You don't have to. A staging URL + API spec is enough for the agents to drive your product as a black box. Source repos are optional — they only let us trace failures deeper.
